# Risks

作成日: 2026-06-18

## セキュリティ上の注意点

現リポジトリは実装アプリ本体ではなく資料中心のため、アプリケーションコード由来の脆弱性評価は未実施です。

現時点で確認できる注意点:

- `docs/proposal/` 配下には、業務フロー、権限、DB概念、インフラ構成、見積・WBSなど、社外公開に適さない可能性がある情報が含まれます。
- `docs/proposal/_proposal__exports/` 配下にはPDF/HTMLのエクスポート成果物が含まれ、Markdown原本と同等またはそれ以上に共有されやすいため、公開範囲の確認が必要です。
- `.vscode/settings.json` に個人環境依存と見られる絶対パスが含まれます。秘密情報ではありませんが、環境情報の露出・再現性低下につながります。
- `package-lock.json` は存在しますが、今回 `npm audit` は実行していません。パッケージ更新禁止・ネットワーク制限の前提があるため、依存脆弱性は未確認です。

## 未管理ファイル

Git未管理ファイルとして以下を確認しました。

- `.DS_Store`
- `docs/.DS_Store`

これらはmacOSのメタデータファイルで、通常はリポジトリ管理・共有対象にしません。削除または `.gitignore` 追加の候補ですが、今回のAuditでは既存設定を変更していません。

## 公開してはいけない可能性のあるファイル

簡易探索で、以下のファイル名パターンは確認されませんでした。

- `.env`
- `.env.*`
- 秘密鍵・証明書系: `*secret*`, `*credential*`, `*key*`, `*.pem`, `*.p12`
- DBダンプ・ローカルDB系: `*.sql`, `*.sqlite`, `*.db`
- ログ・アーカイブ系: `*.log`, `*.zip`, `*.tar`, `*.gz`

ただし、ファイル名ベースの確認であり、すべてのファイル内容を秘密情報検査したわけではありません。

## 資料と実装のギャップ

既存資料には Laravel Application、MySQL、ファイルストレージ、メール通知、CSV出力、バックアップなどの構成案が記載されています。一方で、現リポジトリには該当する実装や設定が確認できません。

リスク:

- 開発着手前の資料リポジトリなのか、実装リポジトリが別に存在するのか判断が必要です。
- 資料上の構成を前提に見積・設計・運用判断を進める場合、実装リポジトリや本番/検証環境の所在確認が必要です。
- READMEが空のため、初見の参加者がリポジトリの目的、資料生成方法、成果物の扱いを判断しづらい状態です。

## ドキュメント管理上の注意

- Markdown原本とPDF/HTMLエクスポート成果物が同一リポジトリで管理されています。どちらを正とするか明記されていません。
- エクスポート成果物の再生成手順が未確認です。
- `package.json` に生成用 `scripts` がないため、資料生成の手順が属人化する可能性があります。

## WordPress差分Auditで追加確認した注意点

確認日: 2026-06-18

### 不要プラグイン有効化

`wp-content/plugins` 配下に以下のプラグインファイルを確認しました。

- Advanced Custom Fields
- Contact Form 7
- WP-Members

現時点では有効化状態をDBで確認していません。不要なプラグインを有効化すると、管理画面・フロント表示・メール送信・会員登録・アクセス制限の挙動が変わる可能性があります。

注意:

- 本番/検証環境で一括有効化しない
- 必要な画面・機能との対応関係を確認してから有効化する
- 有効化前にバックアップと復旧手順を確認する

### EC誤稼働

WooCommerce本体プラグインは確認していませんが、テーマ内にはWooCommerce関連テンプレートと `WC()->cart` 参照があります。

- `wp-content/themes/taguchi_system/woocommerce/single-product.php`
- `wp-content/themes/taguchi_system/woocommerce/cart/cart.php`
- `wp-content/themes/taguchi_system/page-cart.php`
- `wp-content/themes/taguchi_system/page-checkout.php`
- `wp-content/themes/taguchi_system/header.php`

ECは現時点ではダミー/保留です。WooCommerce本体や関連設定を不用意に有効化すると、カート、チェックアウト、注文、メール通知、在庫、決済などが意図せず動作する可能性があります。

注意:

- WooCommerce系は主要対象にしない
- 決済、注文メール、在庫連動、会員登録連動を未確認のまま稼働させない
- ECページはダミー/保留として明示して扱う

### 更新事故

追加されたテーマ・プラグインは、WordPress管理画面やWP-CLIから更新可能になる可能性があります。

リスク:

- テーマ更新でテンプレート改修が上書きされる
- プラグイン更新でフォーム、会員、カスタムフィールドの挙動が変わる
- WooCommerce関連コードがあるため、更新によりEC連携の前提が変わる
- 本番で直接更新すると、差し戻しが難しくなる

注意:

- `composer update`、`npm update`、`wp plugin update`、管理画面からの更新は事前検証なしに行わない
- 更新前後の差分、バックアップ、ロールバック手順を用意する
- プラグインの必要性を分類してから更新対象を決める

## Proposal優先順位整理で追加確認したリスク

確認日: 2026-06-18

### 過剰実装

既存Proposal v1には、EC、会員、カート、決済、注文管理、承認、現場別原価管理、管理画面、発注先ポータルなどが含まれます。

現時点で合意可能なMVPは「FAX・電話による発注業務のオンライン化」であり、フォーム入力、メール送信、PDF発注書生成に限定します。

リスク:

- Proposal v1を最新MVPと誤認すると、初期実装が過剰になる
- 見積、WBS、画面設計が現時点の合意範囲を超える
- 意思決定が未確定の機能まで作り込み、手戻りが増える

注意:

- Proposal v1は将来構想・拡張ロードマップとして保持する
- MVP見積とProposal v1見積を分ける
- DBによる本格的な注文管理はMVP必須にしない

### EC誤稼働

EC、会員制、カート、決済、注文管理システムは現時点では対象外です。

リスク:

- テーマ内のWooCommerceテンプレートやカート導線が、MVP対象と誤認される
- WooCommerce本体や関連プラグインが有効化され、カート、注文、メール、決済導線が意図せず動く
- EC前提の画面や文言が残り、現場・本社の運用とずれる

注意:

- EC機能は保留として明記する
- WooCommerce系は主要対象にしない
- カート/チェックアウト/商品詳細/決済はMVPから分離する

### 業務フロー未確認

MVPはFAX/電話の置き換えが主目的ですが、実際の現場発注フローは未確定事項が残っています。

確認が必要な例:

- 誰がフォームを入力するか
- 発注内容の必須項目
- 本社・管理側の受信担当
- メール確認後の社内処理
- PDF発注書の正式性
- 誤送信・訂正・キャンセルの扱い
- FAX/電話を残す例外条件

これらが未確認のまま実装すると、オンライン化しても本社側の確認作業や電話確認が残る可能性があります。

### MVP+1自動印刷

PDF自動保存・自動印刷はMVP必須要件ではなく、FAX運用に近い受け取り体験を維持するための追加案です。

リスク:

- 誤印刷
- 重複印刷
- プリンタ停止時の未検知
- 受注用PCの停止・未ログイン
- 共有フォルダや同期失敗による未印刷
- 印刷済み/未印刷の判別不能

注意:

- サーバーから直接プリントする前提にしない
- 「PDF生成 -> 受注用PCが取得/同期 -> 印刷」の構成案として扱う
- 自動保存先、PC名、共有フォルダ、プリンタ名、印刷部数、失敗時通知は未確定として扱う
- 自動印刷を実施する場合は、重複防止、失敗通知、再印刷手順を先に定義する
